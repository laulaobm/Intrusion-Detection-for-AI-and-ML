\subsection{Training Experience}
\label{sec:dataset}
For this project we are going to use the 'UNSW\_NB15' dataset \cite{UNSW} which can be found on Kaggle. This data set was used to train and test various systems in order to detect hacking or malicious activity. 
The data set was chosen based on the contained features and the fact that it was used in various papers like \cite{10.1016/j.jnca.2024.103925} and \cite{10.1016/j.jksuci.2024.102164}. Furthermore the same dataset can be found in different Kaggle notebooks, which are also covering intrusion and threat detection. The following notebooks use the same dataset:

\begin{itemize}
    \item \href{https://www.kaggle.com/code/getanmolgupta01/unsw-nb15-cybersecurity-threat-detection-ann/notebook}{UNSW\_NB15 | Cybersecurity Threat Detection | ANN} \cite{unsw_nb15_ann_notebook}
    \item \href{https://www.kaggle.com/code/akritiupadhyayks/cyber-attack-detection-with-random-forest}{Cyber Attack Detection with Random Forest} \cite{cyber_attack_rf_notebook}
\end{itemize}


\nextparagraph
The data in the set was collected by the Cyber Range Lab of the Australian Centre for Cyber Security (ACCS). They used the tcpdump tool and captured 100 GB of raw network traffic. The set includes 9 different types of attacks:

\begin{enumerate}
    \item Fuzzers (sending random data to break systems)
    \item Analysis (scanning for weaknesses)
    \item Backdoors (secret hidden access)
    \item DoS (flooding a system to crash it)
    \item Exploits (using known bugs to attack a system)
    \item General attacks (general cryptographic attacks)
    \item Reconnaissance (spying the network)
    \item Shellcode (injecting malicious code)
    \item Worms (self-spreading malware)
\end{enumerate}


\nextparagraph
The data was collected by simulating normal activities (browsing, emails, downloads) and fake attacks.  IXIA PerfectStorm was used to generate realistic network traffic including normal traffic and attacks, which were then recorded via tcpdump. Using Argus \& Bro-IDS the raw data was turned into meaningful features. In total the dataset contains 49 features which are described in UNSW-NB15\_features.csv. The features describe network traffic behavior between two endpoints, and each record represents one connection and describes the following aspects of the communication:

\begin{itemize}
    \item \textbf{Basic connection information}
    \begin{itemize}
        \item Source and destination IP addresses
        \item Source and destination Ports´
        \item Protocol type
        \item Service type
    \end{itemize}
    \newpage
    \item \textbf{Time-based features}
    \begin{itemize}
        \item Connection duration
        \item Start and end time
        \item Time gaps between packages
    \end{itemize}

    \item \textbf{Traffic volume feature}
    \begin{itemize}
        \item Bytes sent and received
        \item Packet counts
        \item Data transfer rates (in bits per second)
    \end{itemize}

    \item \textbf{Packet characteristics}
    \begin{itemize}
        \item Average packet sizes
        \item Packet timing
        \item Packet intervals
    \end{itemize}

    \item \textbf{Reliability and loss metrics}
    \begin{itemize}
        \item Packet loss and retransmissions
        \item Flow stability indicators
    \end{itemize}

    \item \textbf{Network-level indicators}
    \begin{itemize}
        \item Time-To-Live values
        \item Routing related behavior
    \end{itemize}

    \item \textbf{TCP-Specific features}
    \begin{itemize}
        \item TCP window sizes
        \item Sequence numbers
    \end{itemize}

    \item \textbf{Application level features}
    \begin{itemize}
        \item HTTP transaction depth
        \item Response body size
    \end{itemize}
\end{itemize}

\nextparagraph
Each record has a binary label, 0 indicates normal network traffic and 1 marks an attack.
In total there is a number of 540,044 records which are stored in the following files: UNSW-NB15\_1.csv, UNSW-NB15\_2.csv, UNSW-NB15\_3.csv and UNSW-NB15\_4.csv. Each row of these files contains all of the 49 features.

\subsubsection{Attack catgories and recognition}
The dataset can be used to recognize different attacks, using the given features. Each attack type has a specific behavior which can be detected in the network using patterns and a combination of features. In the following we provide some examples on how the attacks could be discovered by the models:


\vspace{4mm}
\nextparagraph
\textbf{DoS (Denial of Service)}

\nextparagraph
Such attacks can often be recognized by the flood of network traffic which is produced during the attack. Taking a look at features like \texttt{sload} and \texttt{dload}, which represent the bits per second could point out extremely high traffic rates. Apart from that, the number of exchanged packages (\texttt{spkts, dpkts}) and the data volume (\texttt{sbytes, dbytes}) are able to indicate a Denial of Service attack. Finally an increased number of packet loss (\texttt{sloss, dloss}) is also a hint for malicious behavior.


\vspace{4mm}
\nextparagraph
\textbf{Reconnaissance (Scanning)}

\nextparagraph
This attack aims to find weaknesses in the network. Therefore, the network needs to be scanned, which could be recognized via very short connections (\texttt{dur}), sending very few packets (\texttt{spkts}), the usage of many different ports (\texttt{dsport}), unknown services (\texttt{service}), or very small intervals of connections (\texttt{sintpkt}).

\vspace{4mm}
\nextparagraph
\textbf{Exploits}

\nextparagraph
The main aim behind such an attack is to use vulnerabilities to gain access to a network.  Some indicators for exploits could be the usage of HTTP, FTP or vulnerable services (\texttt{service}),  very complex and unusual requests (\texttt{trans\_depth}), abnormal response sizes (\texttt{res\_bdy\_len}) or unusual session behavior (\texttt{state}).

\vspace{4mm}
\nextparagraph
\textbf{Worms}

\nextparagraph
Worms are malware and have the characteristic, that they are self-spreading across the machine/network. This property can be used to recognize them. Considering the destination IP addresses (\texttt{dstip}) it could be suspicious if there are many different destinations. Furthermore repeated connections (\texttt{spkts, dpkts}) or short repeated session (\texttt{dur}) are also indicators for worms. Considering the bit rate from the source (\texttt{sload}), a consistent outgoing traffic could also be a indicator for such malware.

\vspace{4mm}
\nextparagraph
\textbf{Backdoors}

\nextparagraph
With backdoors, attackers gain hidden access to a system. To detect this type of attack it could be useful to take a look at long lasting connections (\texttt{dur}), unusual or rare services (\texttt{service}) and persistent connections (\texttt{state}).

\vspace{4mm}
\nextparagraph
\textbf{Fuzzers}

\nextparagraph
Fuzzers are used to send random data to break systems. Looking at irregular size traffic (\texttt{sbytes, dbytes}), inconsistent packet sizes (\texttt{smeanz, dmeansz}) or many failed or abnormal states is important in order to detect fuzzers.

\vspace{4mm}
\nextparagraph
\textbf{Analysis}

\nextparagraph
Considering this attack, the main goal of the hacker is to study the system behavior. Therefore, the system needs to be scanned. Such scanning could be detected by longer network connections (\texttt{dur}) or moderate interaction (\texttt{trans\_depth}).

\vspace{4mm}
\nextparagraph
\textbf{Shell code}

\nextparagraph
In order to detect malicious code in the network, it is necessary to take a look at payload-heavy requests (\texttt{sbytes}), small or unusual responses (\texttt{dbytes}), services like HTTP and unusual payload sizes (\texttt{res\_bdy\_len}).

\vspace{4mm}
\nextparagraph
\textbf{Generic Attacks}

\nextparagraph
Discovering cryptographic attacks or brute-force attacks can be done with looking at the following properties: high consistent traffic (\texttt{sload, dload}), many repeated attemps (\texttt{spkts}), repetitive sessions (\texttt{dur}) and repeated connection attempts (state). 


\subsection{Learning Task}

In the context of this project, the fundamental learning task for our Machine Learning models is framed as a network traffic classification problem. Based on the features extracted from the dataset, the models must learn the underlying patterns and boundaries that separate benign user activity from malicious network intrusions. 

\nextparagraph
To comprehensively evaluate the capabilities of our chosen algorithms (K-means, SVM, and Random Forest), we divide the learning task into two distinct classification objectives:

\begin{itemize}
    \item \textbf{General Threat Detection (Binary Classification):} The primary task across all three models is to accurately determine whether a given network record represents normal traffic or an attack. The models will take the 49 preprocessed features as input and output a binary label (0 for normal, 1 for attack). This general approach establishes a baseline for how well the models can detect anomalies without needing to understand the specific nature of the threat.
    
    \item \textbf{Attack Categorization (Multiclass Classification):} The secondary, more granular task is to train the models to specifically recognize and differentiate between the 9 distinct attack categories present in the dataset (e.g., DoS, Fuzzers, Worms). This requires the models to learn the specific feature combinations and thresholds—such as payload sizes, connection durations, and port usage—that characterize each unique attack vector as outlined in Section~\ref{sec:dataset}.
\end{itemize}

\nextparagraph
By tackling both a general binary task and a specific multiclass task, we aim to thoroughly assess the flexibility, accuracy, and practical viability of these models within a real-world Intrusion Detection System.

\subsection{Performance Measure}
To evaluate the performance of the trained models, we consider different metrics that have been already applied by researcher in numerous papers concerning intrusion detection. Primarily, we derived our metrics for the comparison from \cite{SVM}

\nextparagraph
First and foremost, the models will be compared to each other with the \textbf{Accuracy}, where the number of successfully recognized anomalies in network traffic is divided by the total number of predictions.

\nextparagraph
With the \textbf{Precision} the number of rightfully identified attacks are compared to the total number of traffic examples classified as an attack to demonstrate the ratio, to which an attack is rightfully classified. To further evaluate the correct recognition of truly positive attacks, the \textbf{Recall} compares the true positives with the number of true positives and false negatives. The trade-off between these to metrics will then be analyzed with the \textbf{F1 score}. 

\nextparagraph
To get more information on falsely classified attacks, the \textbf{False Positive Rate (FPR)} as well as \textbf{Matthews Correlation Coefficient (MCC)}, \textbf{Detection Rate (DR)} and \textbf{False Alarm Rate (FAR)} are considered. With some of these metrics, the potential flaws and limitations of the models should be derivable.

\nextparagraph
Our goal is to reach comparable results with our models to the papers that inspired our approaches, since our goal is to compare the approaches with the same input data respectively. Therefore the \textbf{DR} should be below 0.150 and all other metrics should be $>=$90\%, or better still, over 95\%.
